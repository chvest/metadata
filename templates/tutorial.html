Tutorial: From Document to Receiver

A step-by-step walkthrough of how metadata is attached to a document, what it looks like in two different standards, and what happens at the boundaries where standards differ.

{# ── Scenario setup ──────────────────────────────────────────────────────── #}
Scenario
A NATO analyst at HQ AIRCOM produces an interoperability assessment report. She attaches metadata in {{ src_std.name }} — the NATO standard for resource metadata. The report is then shared with a partner organisation whose catalogue system expects {{ tgt_std.name }} metadata. This tutorial traces what happens at each step.

Metadata is attached — {{ src_std.name }}

The analyst's NATO system creates a structured metadata record conforming to {{ src_std.name }} ({{ src_std.full_name }}). This standard is designed for NATO's information management environment and includes fields that have no equivalent in civilian catalogue standards — most notably the confidentiality labels.

Why these fields? The two confidentiality label fields (metadataConfidentialityLabel and originatorConfidentialityLabel) are mandatory in {{ src_std.name }}. They encode the NATO security policy, classification, and releasability in a structured machine-readable format defined by ADatP-4774. This is what allows NATO systems to enforce access control automatically.

⚠ Three date fields collapse into one

{{ src_std.name }} distinguishes dateCreated, dateIssued, and dateModified as separate fields. Dublin Core has a single generic date element. All three map to dc:date, meaning the converted record contains three date values with no way to tell which is which. A receiver relying on dc:date for sorting by publication date may pick the wrong value.

⚠ Structured contact is flattened to free text

In {{ src_std.name }}, creator is a structured PointOfContact with machine-readable sub-fields for name, role, affiliation, email, and phone. Dublin Core dc:creator is a plain text string. The conversion produces a readable label, but the individual sub-fields — including the email address — are no longer machine-actionable. A system that wanted to automatically notify the author cannot do so from the Dublin Core record alone.

{# ═══════════════════════════════════════════════════════════════════════════ STEP 6 — What this means for the receiver ═══════════════════════════════════════════════════════════════════════════ #}
6

What this means in practice

⛔ Cannot enforce access control

The security and releasability labels are silently dropped. The receiving catalogue has no machine-readable indication that this document is NATO UNCLASSIFIED — Releasable to NATO nations. Automated access control policies based on the metadata cannot be applied; a human must manually review and re-classify the record.

⚠ Records management is lost

The recordsDisposition field — which tells the system to retain the document for 5 years and review in 2029 — has no Dublin Core equivalent and is dropped entirely. The receiving system has no automated way to enforce the retention schedule.

✓ Discovery still works

Core discovery fields — title, description, subject keywords, language, format, and identifier — all convert cleanly. The partner catalogue can find and display the document correctly. The identifier (urn:nato:resource:2024:REP-AGS-0047) is preserved exactly, so a link back to the NATO system remains valid.

↻ Return trip requires work

If the partner updates the record in their catalogue and sends it back, the security labels and disposition rules are gone and cannot be reconstructed. The returned record would need to be manually re-enriched before re-ingestion into the NATO system.
