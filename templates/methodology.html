{# ═══════════════════ 1. OVERVIEW ═══════════════════ #}

1. Overview

This tool assesses interoperability between metadata standards by asking a concrete question: if you have a metadata record conforming to standard A, how much of it can be expressed in standard B, and what is lost or degraded in the translation?

The assessment has three layers, applied independently:

  1. Field-level crosswalk — for each field in standard A, is there a semantically equivalent field in standard B?
  2. Conflict analysis — are there structural, vocabulary, or obligation incompatibilities that would prevent a fully conformant record from surviving the conversion intact?
  3. Derived metrics — compatibility scores, round-trip fidelity, and conversion paths are all computed mechanically from the crosswalk and conflict data.
The tool does not perform automated parsing or comparison of the standard documents themselves. The crosswalk and conflict data were built by reading each standard's field definitions and hand-coding the semantic relationships. All derived outputs (scores, verdicts, round-trip traces) are computed deterministically from that data.
{# ═══════════════════ 2. FIELD MAPPING ═══════════════════ #}

2. Field mapping (crosswalk)

For each directed pair of standards (A→B), every field in A is examined and assigned one of four mapping types:

Type Definition Example
exact The fields share the same semantic meaning, the same data type, and compatible cardinality. A value can be transferred without modification. ADatP-5636 title → Dublin Core dc:title
similar The fields share the same core meaning but differ in structure, encoding, or cardinality in a way that requires minor adaptation but preserves the essential information. ADatP-5636 creator (structured PointOfContact) → Dublin Core dc:creator (free text) — same concept, different encoding
partial The target field covers the same broad concept but only partially expresses the source field's semantics — either because the target is more generic, or because multiple source fields compete for the same target slot. ADatP-5636 dateCreated, dateIssued, and dateModified all map to dc:date — information about which date is which is lost
none No equivalent field exists in the target standard. The source field has no semantic home in the target namespace. ADatP-5636 metadataConfidentialityLabel → Dublin Core (no equivalent)

How mapping decisions were made

Each mapping was determined by the following process:

  1. Read the field definition (name, description, representation term, cardinality, allowed values) from the source standard's normative text.
  2. Identify candidate fields in the target standard with overlapping semantic intent.
  3. Compare representation terms (e.g. Text vs PointOfContact), cardinality constraints (one vs many), and controlled vocabulary compatibility.
  4. Assign a mapping type based on the criteria above.
  5. Record any semantic differences in the notes field.

Mappings are directional. A→B and B→A are assessed separately. The reverse crosswalk is not assumed to be the mirror image of the forward one — a target standard may have fields that map back to different source fields, or to none at all.

Inferred reverse mappings: When computing round-trip fidelity, if no explicit B→A crosswalk entry exists for a field, a reverse entry is inferred from the A→B entry (same mapping type, same fields in reverse). This means round-trip scores for pairs without a hand-coded reverse crosswalk are approximate. Inferred entries are marked as such in the source code.
{# ═══════════════════ 3. SCORES ═══════════════════ #}

3. Compatibility scores

The compatibility score for a directed pair A→B is defined as:

score(A→B) = |{fields in A with a non-none mapping to B}| ÷ |{all fields in A}|

This gives the fraction of A's fields that have some representation in B. It does not weight by importance, obligation level, or mapping quality — a partial mapping contributes the same as an exact one.

Scores are not symmetric. score(A→B) and score(B→A) measure different things: how much of A survives in B, and how much of B survives in A. A high score in one direction does not imply a high score in the other. This asymmetry is intentional and expected — for example, a simple standard will often score higher when sending to a complex standard (fewer fields to map) than when receiving from one (many incoming fields have no home).

What scores do not capture:

  • Whether mandatory fields specifically survive (a score of 40% could mean all mandatory fields map, or none of them do)
  • The quality of mappings (exact vs partial are counted equally)
  • Value-level compatibility — two fields may both be called language but use different controlled vocabularies

For a fuller picture, use the pairwise comparison which shows mandatory-field-specific coverage and the conflict analysis alongside the score.

{# ═══════════════════ 4. CONFLICTS ═══════════════════ #}

4. Conflict identification

Conflicts are interoperability problems that go beyond "the field doesn't exist in the target." They fall into two categories:

4.1 Hand-coded conflicts

{{ stats.n_manual_conflicts }} conflicts were identified through careful reading of both standards' normative texts and manually coded. Each covers a specific structural or semantic incompatibility that would not be apparent from the field mapping alone.

Conflict typeCountDescription
mandatory_gap {{ stats.conflict_type_counts.get('mandatory_gap', 0) }} A field that is mandatory in the source has no mapping to the target — identified manually where the consequences are particularly significant (e.g. the security classification label).
vocabulary {{ stats.conflict_type_counts.get('vocabulary', 0) }} Source and target fields share the same concept but use incompatible controlled vocabularies or encoding schemes. Values cannot be transferred without a lookup table or transformation.
obligation_inversion {{ stats.conflict_type_counts.get('obligation_inversion', 0) }} A field that is mandatory in the target has no guaranteed equivalent in the source — source records may arrive with the required field missing.
structural {{ stats.conflict_type_counts.get('structural', 0) }} A fundamental structural feature of one standard (e.g. a multi-layer security model) has no architectural analogue in the other.
domain_mismatch {{ stats.conflict_type_counts.get('domain_mismatch', 0) }} Two fields appear to map by name or concept but serve different purposes in their respective domains; a mapping would produce misleading results.
4.2 Auto-detected conflicts

In addition to hand-coded conflicts, the tool automatically detects mandatory gap conflicts: any field that is marked mandatory in the source standard and has no crosswalk entry (or a none entry) in the target is flagged as a blocking conflict. These are labelled auto-detected throughout the interface.

Avoiding double-counting: If a mandatory field is already covered by a hand-coded conflict, the auto-detector skips it to avoid duplicates. Hand-coded entries take precedence and typically provide richer explanatory text.
4.3 Severity levels
⛔ Blocking

A conformant source record cannot be made fully conformant in the target without discarding a mandatory field or fabricating a value. The receiver may parse the file but it will fail validation against the target standard.

⚠ Lossy

Conversion is possible and the record will pass target validation, but some information is permanently degraded. The lost structure or precision cannot be recovered on the return trip.

↻ Transform required

Fully interoperable, but only after a specific, well-defined transformation is applied. No information is permanently lost if the transformation is correctly implemented.

Severity is assigned manually for hand-coded conflicts based on the operational consequence of the incompatibility. For auto-detected mandatory gaps, severity is always blocking (a mandatory field with no mapping cannot be preserved).

{# ═══════════════════ 5. VERDICT ═══════════════════ #}

5. Interoperability verdict

The verdict synthesises the score and conflict data into a single label for each pair. It is computed by the following decision tree, applied in order:

  1. Same suite — if both standards belong to the same suite (i.e. are explicitly designed to work together), the verdict is "Designed to work together" regardless of score or conflicts.
  2. Highly interoperable — no blocking conflicts, no lossy conflicts, and the weaker directional score ≥ 50%.
  3. Interoperable with transformation — no blocking or lossy conflicts, but transform-required conflicts exist or the weaker score ≥ 25%.
  4. Partial data loss — lossy conflicts exist but no blocking conflicts.
  5. Not fully interoperable — blocking conflicts exist, or the stronger directional score < 25%.
The verdict is a coarse summary. A pair can receive "Not fully interoperable" even if 95% of fields convert perfectly — one blocking conflict is sufficient. Always read the conflict detail to understand which fields are affected and whether they are relevant to your use case.
{# ═══════════════════ 6. ROUND-TRIP ═══════════════════ #}

6. Round-trip fidelity

Round-trip fidelity simulates what happens to each field in standard A if a record is converted A→B and then converted back B→A. For each field, the tool traces two legs:

  1. Leg 1 (A→B): Look up the A→B crosswalk entry for this field.
  2. Leg 2 (B→A): If leg 1 found a target field in B, look up the B→A crosswalk entry for that B field.

The outcome for each field is classified as:

  • Survives — both legs are exact or similar. The field comes back with the same semantic content.
  • Degraded — at least one leg is partial. The field returns but with reduced precision or structure.
  • Lost — leg 1 maps to none, or leg 1's target field has no reverse mapping in leg 2.
Inferred reverse legs: Where no explicit B→A crosswalk entry exists, the reverse is inferred by treating the A→B entry backwards (same field names, same mapping type). This means the round-trip result for asymmetric pairs is an approximation — it assumes the gateway system would use the same field for both directions, which may not hold in practice.
{# ═══════════════════ 7. CONVERSION PATHS ═══════════════════ #}

7. Conversion paths

Conversion path analysis answers the question: can standard A reach standard C, either directly or via an intermediate standard B?

A directed edge exists from standard X to standard Y if at least one non-none crosswalk entry exists in the X→Y direction. The tool finds:

  • Direct paths (one hop): A→C
  • One-intermediate paths (two hops): A→B→C for every B in the set of standards

Each path is scored by the weakest leg — the minimum of the compatibility scores for each hop. This "bottleneck fidelity" reflects the real-world constraint that a chain is only as strong as its weakest link.

Paths are rated: good (≥ 40%), fair (≥ 20%), or poor (< 20%).

Paths of more than two hops are not computed. In practice, each conversion hop compounds information loss, making three-hop paths rarely useful even if they exist.
{# ═══════════════════ 8. EVIDENCE BASIS ═══════════════════ #}

8. Evidence basis

Each standard pair is assigned one of three evidence levels depending on whether authoritative published crosswalk documentation exists:

LevelMeaningExamples
Normative The mapping is defined by one of the standards themselves — no interpretation is required. DCAT-AP ↔ Dublin Core (DCAT-AP uses the dct: namespace throughout its own specification); INSPIRE ↔ ISO 19115 (INSPIRE TG Annex C maps every element to ISO 19115 by clause)
Published An authoritative body has published a crosswalk document that defines the mapping, though it is external to the standards themselves. DCAT-AP / INSPIRE ↔ ISO 19115 via GeoDCAT-AP 2.0.0 (EU ISA² programme, Appendix B)
Assessment No public crosswalk exists. The mapping is based on our own semantic analysis of the field definitions in both standards. All pairs involving ADatP-5636 — NATO NSO documents are access-restricted and no public crosswalk has been published

The evidence level is shown on every pairwise comparison page and in every compatibility report. Findings with only an assessment basis should be treated as indicative — they represent informed judgement, not normative fact.

Source links and relevant section references are provided throughout the tool so that every claim can be checked against the primary documents.

{# ═══════════════════ 9. LIMITATIONS ═══════════════════ #}

9. Limitations

The following aspects are out of scope for this tool:

  • Value-level validation — the tool checks whether a field has a mapping, not whether a specific value in that field would survive conversion. For example, a NATO security classification value ("NATO SECRET") has no equivalent in Dublin Core's dc:rights free-text field, but this is only partially captured in the conflict notes, not enforced at record validation time.
  • Implementation correctness — the tool assesses conceptual interoperability between the specifications. Whether a specific software system correctly implements that specification is a separate question.
  • Profiles and extensions — many standards are deployed as profiles (e.g. a national DCAT-AP profile that adds mandatory fields). The tool covers the base standards only; local profiles may have different compatibility characteristics.
  • Three-or-more-hop paths — conversion chains longer than one intermediate standard are not analysed.
  • Restricted NATO documents — ADatP-5636, ADatP-4774, and ADatP-4778 are NATO NSO documents. The field definitions used in this tool are based on publicly available NATO standardisation information and published NATO policy; some nuances of the normative text may not be fully reflected.
  • Standard evolution — standards are versioned documents. The versions covered are listed in each standard's description. Newer versions of any standard may change field definitions, obligations, or vocabulary encodings in ways not yet reflected here.

10. Replication guide

All data that drives the assessments is in two files in the standards/ directory:

FileContainsTo critique or extend
standards/crosswalk.py All CrosswalkEntry objects (the hand-coded field mappings) and all ConflictEntry objects (hand-coded conflicts), plus the PAIR_SOURCES verification sources. Add, modify, or remove CrosswalkEntry or ConflictEntry objects. All scores, verdicts, round-trip results, and conversion paths recompute automatically on next request.
standards/nato/adatp5636.py
standards/eu/dublin_core.py
(etc.)		 Field definitions for each standard — name, description, obligation, cardinality, representation term, allowed values, and section reference. Modify FieldDefinition objects to correct or extend field definitions. The reference attribute holds the section citation shown throughout the UI.

To add a new standard, create a new file in the appropriate subdirectory, define a class inheriting from MetadataStandard, register it in standards/registry.py, and add crosswalk entries for it in standards/crosswalk.py. The UI, scores, and reports will pick it up automatically.

Testing changes: After editing Python files, restart the Flask server (Ctrl+C then run.bat). The test client at the bottom of app.py checks that all standard pages return HTTP 200. Run it with python -c "from app import app; ..." as shown in the developer notes.
{# ═══════════════════ 11. RELATED ACADEMIC LITERATURE ═══════════════════ #}

11. Related academic literature

The following peer-reviewed publications address metadata standard comparison, crosswalk methodology, and interoperability in ways that are directly relevant to the approach used in this tool. No single publication compares the exact combination of standards covered here (NATO ↔ EU ↔ ISO ↔ NIST), but the methodological foundations — crosswalk design, compatibility scoring, schema-level vs record-level interoperability — are well established in the library and information science literature.

{# Publication cards #}
Metadata Interoperability and Standardization — A Study of Methodology (Parts I & II)
Chan, L.M. & Zeng, M.L.  ·  D-Lib Magazine, Vol. 12, No. 6, June 2006  ·  DOI: 10.1045/june2006-chan
Foundational

The most widely cited treatment of metadata interoperability methodology. Examines how rapid growth of Internet resources and digital collections has produced a proliferation of metadata schemas across different user communities and resource types. Identifies three operational levels where interoperability can be achieved: schema level (crosswalks, application profiles, registries), record level (element-by-element semantic mapping to convert or combine metadata records), and repository level (mapping value strings for cross-collection searching). The crosswalk approach and the distinction between structural and value-level interoperability used in this tool are directly grounded in this framework.

Towards an Interoperability Framework for Metadata Standards
Nilsson, M., Johnston, P., Naeve, A. & Powell, A.  ·  Proc. International Conference on Dublin Core and Metadata Applications (DC-2006), Manzanillo, Mexico  ·  DOI: 10.23106/dcmi.952108357
Foundational

Proposes a conceptual metadata framework — built on the DCMI Abstract Model — that integrates metadata vocabularies, schemas, formats, and application profiles into a single unified structure for analysing and comparing standards. The framework is used to compare Dublin Core and IEEE LOM, highlighting key structural differences and gaps. Directly relevant to the field-level mapping approach and the treatment of application profiles (DCAT-AP-SE, INSPIRE) as constrained profiles of base standards used here.

GeoDCAT-AP: Representing Geographic Metadata by Using the DCAT Application Profile for Data Portals in Europe
Perego, A., Cetl, V., Friis-Christensen, A. & Lutz, M.  ·  JRC Technical Report, European Commission Joint Research Centre, 2017  ·  Report No.: JRC107410  ·  JRC Publications
Published crosswalk

The authoritative technical report defining GeoDCAT-AP — an RDF-based representation of ISO 19115 geospatial metadata that is conformant with DCAT-AP. Defines normative mappings between ISO 19115 / INSPIRE metadata elements and DCAT-AP properties, covering classes, properties, cardinality constraints, and controlled vocabulary alignment. This document is the primary source used in this tool for the INSPIRE ↔ ISO 19115 and DCAT-AP ↔ ISO 19115 pair assessments.

GeoDCAT-AP: Use Cases and Open Issues
Perego, A., Friis-Christensen, A. & Lutz, M.  ·  Smart Descriptions & Smarter Vocabularies (SDSVoc), W3C Workshop, 2016  ·  W3C SDSVoc 2016
Published crosswalk

Examines practical problems encountered when designing GeoDCAT-AP, focusing on challenges with potential relevance beyond the geospatial domain: profile-based content negotiation, publishing metadata on the Web, representing API-based data access in metadata, and approaches to modelling data quality. Relevant to the conversion path and round-trip fidelity concepts used in this tool, and to the treatment of INSPIRE as a profile of ISO 19115.

Interoperability of Open Science Metadata: What About the Reality?
Dang, V.N., Aussenac-Gilles, N., Megdiche, I. & Ravat, F.  ·  Research Challenges in Information Science (RCIS 2023), Springer LNBIP, 2023  ·  DOI: 10.1007/978-3-031-33080-3_28
Critical analysis

Assesses the real-world state of metadata interoperability across open science datasets from different research communities (medical and Earth system domains). Finds that the diversity of metadata schemas does not allow native interoperability, and evaluates whether current automated matching tools are sufficiently efficient to bridge this gap. The paper's conclusion — that declared interoperability and practical interoperability often diverge — is consistent with this tool's approach of distinguishing structural field mappings from value-level compatibility.

Metadata Standards for Semantic Interoperability in Electronic Government
Valentim, N., Pacheco, R. & Pinto, H.S.  ·  Proc. 2nd International Conference on Theory and Practice of Electronic Governance (ICEGOV 2008), Cairo, Egypt  ·  DOI: 10.1145/1509096.1509111
Government context

Examines how metadata standards can support semantic interoperability specifically in the e-government domain — a context directly analogous to NATO/EU cross-portal data sharing. Discusses how different public-sector communities adopt different metadata standards and the practical challenges of achieving interoperability across organisational and national boundaries. Provides useful framing for the NATO ↔ EU interoperability problems identified in this tool.

Gap in the literature: No published academic paper was found that specifically compares NATO metadata standards (ADatP-5636, ADatP-4774, ADatP-4778) with EU or ISO standards. The NATO NSO source documents are restricted, which limits public academic engagement with these standards. The assessments in this tool for NATO ↔ non-NATO pairs are therefore classified as assessed (our own semantic analysis) rather than normative or published.
